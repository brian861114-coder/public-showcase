# 東京旅程入口與導覽模板設計

## 理解摘要

- 優化範圍只限 `github-pages` 版本，不修改專案外層的另一版本。
- 主要使用者是第一次去日本的家人。
- 使用者需要按自己的目的，快速找到交通、景點、行前準備及其他工具。
- 目前主要問題是按鈕與入口太多，非設計者不容易理解有哪些功能或如何使用。
- 介面以手機優先；複雜地圖與路線頁也必須適合桌面電腦。
- 視覺方向為日系清爽，並以清晰、易讀、低學習成本為優先。
- 第一階段建立不含真實旅遊內容的候選模板，選定方向後再整合正式網站。

## 假設與非功能需求

- 候選模板放在獨立預覽目錄，不影響現有正式頁面。
- 模板保留必要的功能名稱和用途說明，但不放入真實行程、景點、班次或費用。
- 不引入框架、外部字體、追蹤程式或網路依賴。
- 維持純靜態檔案及 GitHub Pages 相容性，也可直接從本機開啟。
- 使用者規模是家庭小群體；資料維持在目前裝置，不新增個資收集。
- 重視快速載入、離線可理解、低維護成本與基本無障礙。
- 第一階段不重寫交通地圖、預算或其他功能內容。

每套方案包含首頁與一個功能內頁，用來比較入口資訊架構、跨頁導覽及桌面／手機版行為。共用檔案負責基本視覺規則、無障礙與預覽切換；各方案保留獨立版面與導覽邏輯。

## 候選方案

### A：需求問答式（推薦）

首頁用「現在想找什麼？」開場，顯示查看行程、規劃交通、尋找景點、做好準備四個大型任務入口。每個入口包含動詞、用途說明與圖示。次要功能放入「其他工具」。

功能內頁在頂部顯示返回首頁與目前頁面，手機底部保留五個主要入口，內容尾端提供推薦下一步。此方案最適合第一次使用者，但進階功能會藏得稍深。

### B：分類儀表板式

首頁按「旅行中常用」、「出發前準備」、「需要協助」等分類呈現功能。每區先顯示二至三個主要項目，其餘項目可展開。

功能內頁顯示目前所屬分類，底部只保留常用入口。此方案讓功能較容易被發現和擴充，但第一眼選項仍比 A 多。

### C：旅程階段式

首頁按「出發前」、「旅行途中」、「需要協助」分階段，預設展開旅行途中，使用者可切換階段。

功能內頁提供返回目前階段的入口，底部使用「首頁、今天、移動、探索、協助」等情境語言。此方案的旅行情境感最強，但跨階段工具的歸類可能較不明確。

## 共通互動與視覺規則

- 暖白背景、墨色文字、低飽和綠或藍作為重點色。
- 主要操作不依賴隱藏手勢或長按。
- 圖示必須搭配文字；目前位置不只用顏色表示。
- 手機點擊區至少約 44×44px，底部導覽避開裝置安全區域。
- 桌面版將手機底部導覽轉換成適合寬螢幕的側邊或頂部導覽。
- 緊急資訊保持明顯，但避免使用大面積警示紅色。
- 每頁提供「返回方案比較」入口。
- 動畫保持輕微，並支援 `prefers-reduced-motion`。
- 關閉 JavaScript 後，核心連結仍可使用。
- 空白或未實作內容提供簡短說明與下一步，不放置無反應按鈕。

## 驗收方式

三套模板使用相同任務測試：

1. 找到交通路線頁面。
2. 找到景點資訊。
3. 找到行前準備工具。
4. 從功能內頁返回首頁。
5. 從功能內頁切換到另一類工具。
6. 找到緊急協助入口。

第一次看到頁面的使用者應能在沒有口頭說明的情況下指出正確入口。主要任務最好一次點擊即可進入，次要工具不超過兩次點擊。

技術驗收：

- 360px 寬度無水平溢出或文字截斷。
- 可點擊區域至少約 44×44px。
- 鍵盤焦點順序合理，所有連結與按鈕可操作。
- 文字與背景具備足夠對比。
- 無外部資源依賴，核心導覽不依賴 JavaScript。
- 桌面版具有對應版面，而非單純放大手機介面。
- 支援減少動態效果設定。

## 決策紀錄

1. **範圍：**只針對 `github-pages`。外層版本不在本次工作範圍。
2. **交付方式：**先新增獨立候選模板，不直接取代正式頁面，方便安全比較。
3. **比較維度：**三套方案採不同資訊架構，而不是只改配色。
4. **展示深度：**每套包含首頁與功能內頁，以驗證跨頁導覽。
5. **技術方向：**維持無框架的靜態 HTML、CSS 與少量漸進增強 JavaScript。
6. **推薦方案：**A 需求問答式最符合第一次使用者，但三套皆保留供實際選擇。
7. **驗收優先順序：**先驗證理解度與導覽效率，再整合真實內容與功能。
8. **使用者選擇：**正式方向採混合式導覽；入口使用 A 的需求問答式，進入分類後使用 C 的旅程階段式細分。
